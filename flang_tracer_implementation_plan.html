00 Project overview & key insight
The core differentiator
No existing tool traces a Fortran construct across Flang's full 5-stage lowering pipeline. Godbolt exposes LLVM IR only. opt-viewer is LLVM-pass-centric. This tool introduces construct-centric, cross-stage provenance — the user selects a source construct and instantly sees every representation it has from parse tree to machine IR, with annotated cross-references. This is a Flang developer force-multiplier.
01 System architecture Foundation
A · Instrumentation Layer
C++ hooks injected at each Flang pass boundary. Captures serialised representations with source-location metadata. Runs inside a patched flang-new binary via MLIR pass instrumentation API.
B · Correlation Engine
Post-process pass that aligns captured stage dumps by source location, symbol name, and structural fingerprint. Produces a unified TraceBundle — one object per source construct with 5 stage slots.
C · CLI + Renderer
Python CLI wrapping the patched compiler. Drives compilation, collects stage dumps, invokes the correlation engine, then renders to text / annotated HTML / structured JSON.
# High-level data flow Source .f90 → flang-new --ftrace-pipeline (patched binary) → ParseTreeHook → stage0.pt.json → SemanticsHook → stage1.sem.json → HLFIRHook → stage2.hlfir.mlir → FIRHook → stage3.fir.mlir → LLVMIRHook → stage4.ll → CorrelationEngine (SrcLoc + SymHash alignment) → trace_bundle.json → Renderer → output.html / output.json / stdout

02 4-week phased implementation plan Core
Phase 1 — Infrastructure & Parse Tree Hook
Week 1 · Days 1–5
  • Fork LLVM/Flang at HEAD, establish reproducible build environment (CMake + Ninja), pin commit SHA
  • Read and annotate flang/lib/Parser/, lib/Semantics/, lib/Lower/ source trees
  • Implement ParseTreeDumper visitor that serialises each top-level construct to JSON with SourceRange metadata
  • Add --ftrace-pipeline driver flag; wire to dump directory mechanism
  • Write unit tests for dumper on 3 trivial constructs (assignment, DO loop, PRINT)
  • Validate round-trip: parse → dump → load and compare symbol names
Phase 2 — Semantics + HLFIR + FIR Hooks
Week 2 · Days 6–10
  • Extend SemanticsChecker to record per-construct symbol table snapshots and type decorations into stage1.sem.json
  • Register MLIR pass instrument using PassInstrumentation API; capture HLFIR op tree after hlfir-to-fir boundary
  • Capture FIR op tree after fir-to-llvm boundary, extracting per-operation source location attributes
  • Design and implement SourceLocKey — normalised (file, line, col) hash used as cross-stage linkage key
  • Smoke test all three hooks on array_assign.f90 demo file; verify non-empty output at every stage
Phase 3 — LLVM IR Hook + Correlation Engine
Week 3 · Days 11–15
  • Implement LLVMIRExtractor: after FIRToLLVMPass, walk LLVM module and map !dbg / !flang.srcrange metadata to source ranges; emit stage4.ll snippets
  • Build Python CorrelationEngine: load all 5 stage files, perform primary alignment by SourceLocKey, secondary alignment by symbol-name fuzzy match
  • Handle split constructs (one source line → multiple HLFIR ops) with 1-to-N grouping in TraceBundle
  • Implement CrossRefAnnotator: inserts bidirectional [→ FIR op#42] inline markers into stage text representations
  • Integration test: full pipeline on 5 of the 10 demo constructs; correlation accuracy target ≥ 85%
Phase 4 — CLI, Rendering & Demo Suite
Week 4 · Days 16–20
  • Build Python CLI (ftrace) with subcommands: trace, show, diff, export; use click + rich
  • Implement annotated HTML renderer: 5-pane tabbed view, syntax-highlighted each stage, cross-ref hyperlinks, collapsible sections
  • Implement JSON schema output for programmatic consumption + text/plain renderer for CI pipelines
  • Complete all 10 demo construct traces; write lowering-pattern commentary for each in Markdown
  • Write documentation: developer guide, lowering patterns reference, architecture notes; package as GitHub repo with README + examples/
  • Record short screen-capture demo walkthrough showing DO CONCURRENT and WHERE block traces end-to-end

Why this ordering matters for judges
Starting with C01 (trivial) lets reviewers verify correctness with ground truth. C05 (DO CONCURRENT) and C07 (polymorphism) are the two constructs Flang developers most struggle to trace manually — these are the headline demos. C08 (coarrays) is the stretch goal that would be novel even to LLVM committers.

07 Judge-impression differentiators Wow factors
Interactive HTML cross-reference
Clicking any node in any pane highlights the corresponding construct in all other stages. Implemented via shared anchor IDs and a tiny JS highlight engine. No external dependencies.
ftrace diff — two-version comparison
ftrace diff old.f90 new.f90 --construct C05 shows how a code change affects the lowering of DO CONCURRENT at every stage. Useful for Flang contributors evaluating optimisations.
MLIR metadata preservation patch
The project patches FIRToLLVMPass to emit !flang.srcrange metadata into LLVM IR — a novel contribution that could be upstreamed to LLVM. This is a genuine compiler improvement, not just tooling.
Lowering complexity scoring
Each traced construct gets a numeric "lowering expansion ratio" (LLVM IR lines / Fortran source lines). Surfaced in the CLI as a quick complexity signal. Helps Flang devs spot unexpectedly expensive constructs.
Pytest-driven regression suite
All 10 demo constructs have golden-output JSON files. CI runs ftrace trace on each and compares. A Flang update that changes lowering is immediately visible. Contribution-ready for Flang's test infra.
Contribution-ready Flang docs PR
The lowering-patterns reference is formatted to match Flang's existing docs tree and submitted as a draft PR to llvm/llvm-project. Even if not merged, demonstrates real community engagement.
08 Technical risks & mitigations Risk
R1 · Source location metadata loss
MLIR passes may strip loc() attributes, breaking stage correlation. Mitigation: add -mlir-print-debuginfo to all HLFIR/FIR passes; if lost, fall back to symbol-name alignment.
R2 · Flang HEAD instability
Flang is under active development; APIs shift. Mitigation: pin to a tested commit SHA on Day 1; use git log --oneline --since=1.month to identify risky recent changes before pinning.
R3 · Coarray lowering incomplete
Coarray support in Flang is partial. Mitigation: if C08 (coarrays) can't be fully traced, document the gap explicitly and substitute an allocatable derived-type trace — more valuable than a broken coarray trace.
R4 · Build time (LLVM full build ~60 min)
Patching and rebuilding Flang repeatedly is slow. Mitigation: use -DLLVM_TARGETS_TO_BUILD=X86, -DCMAKE_BUILD_TYPE=RelWithDebInfo, and ccache to reduce incremental rebuild to under 5 min.
R5 · 1-to-N construct splitting
A single WHERE block expands to many HLFIR operations. Mitigation: group by a common !flang.construct_id attribute added in the hook; if the attribute is missing, group by enclosing function + line range.
R6 · Python / C++ ABI boundary
Driving the patched binary from Python adds process-launch overhead. Mitigation: batch all 10 demo traces in a single ftrace batch invocation; avoid per-stage re-compilation.
09 Agent execution instructions Must read
Start here — agent bootstrap sequence
1. Clone llvm/llvm-project at the pinned SHA documented in BUILD.md.
2. Build with flags in cmake/flang-tracer.cmake — do NOT use default flags.
3. Run make check-flang to confirm baseline before any patches.
4. Apply patches in order: patches/0001-parse-tree-hook.patch → 0002-semantics-hook.patch → 0003-hlfir-fir-hooks.patch → 0004-llvm-metadata.patch.
5. Each patch has a corresponding test: run pytest tests/test_stage{N}.py before proceeding to the next patch.
6. If a test fails, consult docs/troubleshooting.md before modifying patch content.
